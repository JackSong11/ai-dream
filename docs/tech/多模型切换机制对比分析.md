# 多模型切换机制对比分析：nanobot vs dream

> 本文聚焦一个核心差异——**「对话过程中如何动态切换模型」**，对比 nanobot（Python）与 dream（本项目，Java + Spring AI）两套方案的设计取舍，并结合代码与场景举例说明。

---

## 1. 一句话概括差异

| 维度 | nanobot | dream（本项目） |
|---|---|---|
| 切换的本质 | 替换一个**不可变运行时值**（`LLMRuntime`）的引用 | 用 **modelKey 查表**换一个预建的 `ChatClient` |
| 切换粒度 | 每轮对话开始时从 Resolver 取当前 runtime | 每次请求传 modelKey 从 Registry 取 client |
| 运行时不可变值 | ✅ `LLMRuntime`（frozen dataclass） | ❌ 无，直接持有 `ChatClient` |
| 单次调用覆盖（不改默认） | ✅ `resolve_override()` | ❌ 无（传 key 即当次生效，但无「默认/覆盖」概念） |
| 配置指纹判定是否重建 | ✅ `snapshot_signature` | ❌ 无，`refresh()` 全量重建 |
| 切换入口 | 预设切换 / SDK 单次覆盖 / WebUI 三入口 | 请求携带 modelKey 单一入口 |
| 构建时机 | 切换时按需构建/复用 provider | 启动时一次性预构建全部并缓存 |

---

## 2. 两种模型的心智图

### 2.1 dream：预建缓存 + 查表切换（一次性预热）

启动阶段，`ChatModelRegistry.refresh()` 遍历配置里所有 `provider → models`，把每个模型都构建成 `ChatClient` 塞进 `clientCache`：

```
应用启动
  └─ refresh()
       ├─ 遍历 providers
       │    └─ 遍历 models
       │         └─ factory.create(provider, model) → ChatClient
       │              └─ clientCache.put(modelKey, client)
       └─ 计算 defaultModelKey (primary > defaultModel > firstKey)

运行期每次请求
  └─ registry.getClient(modelKey)   // 纯查表，O(1)
       └─ 拿到已经构建好的 ChatClient 直接用
```

**特点**：所有模型在启动时就全部"预热"好了，切换只是换一个 Map 的 key。简单、快、无状态竞争，但**没有"当前选中模型"这个运行时概念**——每次调用都必须显式带上 modelKey，谁都不持有"默认对话正在用哪个"这一状态（除了兜底的 `defaultModelKey`）。

### 2.2 nanobot：不可变运行时值 + Resolver 持有引用（有"当前状态"）

nanobot 引入了一个"当前选中的运行时"概念，由 `ModelRuntimeResolver` 持有一个 `LLMRuntime` 引用：

```
ModelRuntimeResolver (常驻)
  └─ 内部持有 current: LLMRuntime   // 不可变值，代表"现在默认用这个"

一次对话 turn
  └─ resolver.current()  → 返回当前 LLMRuntime
       └─ runtime.provider.chat_with_retry(...)

切换 /model fast
  └─ resolver.select_preset("fast")
       └─ 构建新的 LLMRuntime
       └─ 替换 resolver 内部引用 (旧的丢弃)
  → 下一轮 turn 自动生效
```

**特点**：有一个"当前默认模型"的持久状态，切换 = 原子替换这个不可变引用。因为是 frozen dataclass，切换过程不会出现"读到一半旧值一半新值"的竞争。

## 3. 逐点详解差异

### 3.1 差异一：有没有"运行时不可变值"

**dream**：业务层直接拿 `ChatClient`（一个可变的、由 Spring AI 构建的对象）去发请求。切换靠换 client 引用。

**nanobot**：把"这一轮要用的所有执行值"——provider 实例、model 名、生成参数、context_window、preset 名、配置指纹——一次性冻结进 `LLMRuntime`。

> 为什么要冻结？因为 provider 对象本身是有状态、可变的。如果消费者一会儿读 `provider.model`、一会儿读 `provider.generation`，在切换发生的瞬间就可能读到不一致的组合。`LLMRuntime` 把这些值**拷贝快照**下来，消费者只认 runtime 字段，杜绝了"provider 变了但某个字段还没变"的中间态。

**举例**：假设一轮对话里工具准入层要读 `context_window`、对话循环要读 `model`。
- nanobot：两者都从**同一个 `LLMRuntime` 快照**读，保证是同一次选择的结果。
- dream：这个问题不存在也不解决——因为 dream 每次请求都是"取一个完整 client 直接用完"，没有跨组件共享"当前选择"的中间态。这正是 dream 更简单的原因，也是它没有这层抽象的原因。

### 3.2 差异二：有没有"单次调用覆盖"

**nanobot** 的 `resolve_override()` 支持：本次请求临时用某个 model/preset，**但不改变默认**。

```python
# 伪代码：SDK 调用
agent.process("你好", model="gpt-4o")   # 仅这次用 gpt-4o
agent.process("再见")                    # 又回到默认预设
```

**dream** 目前没有"默认 vs 覆盖"的分层。传 modelKey 时：

```java
// 每次都得显式带 key，且没有"临时覆盖后自动回退默认"的语义
registry.getClient("kimi").prompt("你好").call();   // 这次用 kimi
registry.getClient(null).prompt("再见").call();      // null → defaultModelKey 兜底
```

区别在于：nanobot 有"会话级默认预设"这个持久状态，`/model` 切换会**长期改变**默认；而单次 override 是**临时**的、用完即弃。dream 只有"全局 defaultModelKey + 每次请求显式指定"两态，缺少"改变会话默认"和"仅本次覆盖"的区分。

**举例场景**：用户在 WebUI 里用 `/model fast` 把整个会话切到快速模型（长期生效），但某条消息想临时用强模型跑一次（单次生效）。
- nanobot：`select_preset("fast")` 改默认 + 某次 `override(model="strong")` 临时用一次，天然支持。
- dream：需要业务层自己记住"会话当前默认 key"，并在每次请求手动传递，框架不提供这层。

### 3.3 差异三：有没有"配置指纹"决定是否重建

**dream** 的 `refresh()` 是**全量重建**：`clientCache.clear()` 后重新构建所有 client。哪怕只改了一个模型的温度，也会把全部 client 推倒重来。

**nanobot** 用 `snapshot_signature`（配置指纹）判断：只有当某个模型的配置真正变化时，才重建对应 provider；未变的复用旧实例，避免不必要的重连开销。`select_model` 甚至用 `dataclasses.replace()` 在**同一个 provider 实例**上只换模型名。

**举例**：配置里有 10 个模型，用户只改了第 3 个的温度并触发热更新。
- dream：10 个 `ChatClient` 全部重建。
- nanobot：签名对比后，只有第 3 个的 provider 需要重建，其余 9 个复用。

> 在 dream 场景下（模型数量有限、`ChatClient` 构建轻量、不常热更新），全量重建是完全可接受的简化；nanobot 面向 30+ 厂商、可能频繁热切换，才需要指纹优化。

### 3.4 差异四：切换入口数量

- **nanobot**：`/model` 聊天命令、SDK 参数覆盖、WebUI/设置 API 三条入口，且通过事件总线（`runtime_model_changed`）通知前端同步。
- **dream**：单一入口——请求携带 `modelKey`。切换即"下次请求传不同 key"，无需事件通知（因为没有"当前选中"这个需要同步的持久状态）。

## 4. 完整对比调用链

### dream：一次带 modelKey 的请求

```
业务 Service
  └─ registry.getClient("deepseek-chat")   // 查表，命中缓存
       └─ 返回启动时预建的 ChatClient
            └─ client.prompt(...).call()    // 直接发请求
```

### nanobot：一次 /model 切换后的对话

```
用户 /model fast
  └─ cmd_model 解析
       └─ loop.set_model_preset("fast")
            └─ resolver.select_preset("fast")
                 ├─ 校验预设名
                 ├─ build_provider_snapshot()    // 按需构建/复用 provider
                 ├─ 生成新 LLMRuntime (frozen)
                 └─ 替换 resolver.current 引用
  → 下一轮 turn
       └─ resolver.current() → 新 LLMRuntime
            └─ runtime.provider.chat_with_retry(...)
```

## 5. 设计取舍总结

| | dream 的选择 | 换来的好处 | 潜在代价 |
|---|---|---|---|
| 预建全部缓存 | 启动全量构建 | 运行期切换 O(1)、零构建延迟、逻辑极简 | 启动稍慢、内存常驻全部 client |
| 无运行时不可变值 | 直接持有 ChatClient | 少一层抽象、易理解 | 无法在多组件间安全共享"当前选择"状态 |
| 无单次覆盖 | 每次显式传 key | 语义直白 | 无"会话默认 vs 临时覆盖"分层 |
| 无配置指纹 | refresh 全量重建 | 实现简单 | 热更新时无差别重建（模型少时无所谓） |

**结论**：dream 的方案是**面向"固定、有限的模型集 + 请求级指定"场景的务实简化**——用一次性预建缓存 + modelKey 查表，以最少的抽象层达成"运行时切换"。nanobot 的方案是**面向"海量厂商 + 会话级默认切换 + 热更新 + 容错"场景的完整抽象**——用不可变运行时值隔离状态、用配置指纹优化重建、用多入口统一切换。

如果 dream 未来需要"会话级记忆当前模型 + 临时单次覆盖 + 频繁热更新"，可分别借鉴：
1. 引入一个轻量的"会话模型上下文"持有当前 modelKey（对标 Resolver 的 current）；
2. 提供 `withModel(key)` 的单次覆盖 API（对标 `resolve_override`）；
3. `refresh()` 增加按 modelKey 的配置指纹比对，做增量重建（对标 `snapshot_signature`）。
